## Example 1: Monorepo vs Polyrepo (Engineering Profile)

### Input

```
engineering council: We're building a new platform with 4 microservices, a shared UI 
component library, and a BFF layer. Team of 12 engineers. Should we use a monorepo or 
polyrepo?

Context: New greenfield project. The 4 services share some data models and the UI library
is consumed by 2 frontend apps. Team has experience with Git but not with monorepo tooling
(Nx, Turborepo). Current CI is GitLab.
```

### Phase 0: Framing

The orchestrator identifies:
- **Domain:** Engineering/architecture decision
- **Profile selected:** Engineering (architect, deriver, shipper, systems-mapper, inverter, user-advocate)
- **Stakes:** Long-term development velocity, team coordination overhead, CI complexity
- **Key context:** Shared data models create coupling; team lacks monorepo tooling experience

---

### Phase 1: Independent Advisor Analysis

#### architect (Formal Structure)

> The dependency graph topology is the deciding factor. Shared data models create coupling edges between all 4 services. The shared UI library creates coupling edges to both frontend apps. These edges demand atomic cross-cutting changes.
>
> In a monorepo, these coupling edges are mechanized — one commit propagates changes across all boundaries. In a polyrepo, the same edges become coordination overhead that scales quadratically with the number of shared dependencies.
>
> The formal structure is clear: when components share abstractions (data models, UI contracts), co-location reduces the coordination complexity class from O(N) to O(1). The tooling unfamiliarity is a one-time constant cost; the coordination overhead is a recurring polynomial cost.
>
> **Confidence:** High
> **Where I May Be Wrong:** If the shared data models are actually thin interfaces rather than rich types, the coupling edges may be weaker than the topology suggests.

#### deriver (First Principles)

> Start from what's actually true: code sharing has a cost function. Every shared dependency creates a coupling edge. The question is whether you pay that cost via explicit versioning (polyrepo) or implicit co-location (monorepo).
>
> **The physics here:**
> - 4 services share data models — changes must propagate atomically or you get version skew. Atomic propagation is cheaper in a monorepo (one commit) vs. polyrepo (N publish-bump-update cycles).
> - UI library consumed by 2 apps — same principle. Cross-cutting changes are O(1) in monorepo, O(N) in polyrepo.
> - 12 engineers — small enough that trunk-based development won't create merge hell.
>
> **The actual risk:** monorepo tooling unfamiliarity. But Turborepo/Nx complexity exists to solve *scale* problems you don't have yet with 4 services. A simple `pnpm workspaces` + GitLab CI with path-filtered pipelines gives you monorepo benefits without the tooling tax.
>
> **Verdict:** Monorepo with minimal tooling. The sharing topology demands it. Don't adopt Nx day one — grow into it.
>
> **Confidence:** High
> **Where I May Be Wrong:** If services diverge rapidly in deployment cadence, the "atomic change" benefit erodes and polyrepo's isolation becomes cheaper.

#### shipper (Pragmatic Engineering)

> Monorepo. Here's why:
>
> **4 services sharing data models + shared UI library = constant cross-cutting changes.** In polyrepo, that's 6+ PRs, version bumps, and release coordination for what should be one atomic change. With 12 engineers on a greenfield project, you'll spend more time on dependency management than building product.
>
> **The "no monorepo tooling experience" concern is overrated.** Turborepo takes an afternoon to set up. The learning curve is shallow compared to the ongoing tax of managing 6+ repos, their CI pipelines, versioning strategies, and publish workflows.
>
> **What actually ships:** One repo, Turborepo, GitLab CI with path-based triggers. Shared packages as internal workspace dependencies — no publishing step. Engineers see the whole system, refactors are atomic, onboarding is one `git clone`.
>
> **What breaks in polyrepo:** "Who updated the shared models but forgot to bump the UI lib?" — that 3 AM page.
>
> **Confidence:** High
> **Where I May Be Wrong:** If team ownership splits cleanly along service boundaries within 6 months, polyrepo's isolation becomes worth the coordination tax.

#### systems-mapper (Feedback Loops)

> **Monorepo dynamics:**
>
> A reinforcing loop emerges: shared models in one repo -> atomic cross-cutting changes -> faster integration -> more shared code -> tighter coupling. This loop accelerates delivery initially but creates a balancing loop at scale: build times grow, CI becomes a bottleneck, and a single broken trunk blocks all 12 engineers. The delay between "this feels productive" and "CI is now 40 minutes" is 6-9 months.
>
> **Polyrepo dynamics:**
>
> The dominant loop is drift: independent repos -> versioned contracts -> teams delay upgrades -> schema divergence -> integration bugs -> more defensive coding -> slower delivery. This is a reinforcing decay loop with long delays.
>
> **Leverage point:** The shared data models. Whichever option you choose, the feedback loop around model changes determines system health.
>
> **Confidence:** Medium
> **Where I May Be Wrong:** Teams with strong code ownership norms can break the monorepo coupling loop.

#### inverter (Inversion + Mental Models)

> **Inversion: What guarantees failure?**
>
> 1. **Dependency drift** — shared models diverge across repos, causing silent integration breaks in production.
> 2. **Coordination tax** — cross-repo PRs for a single feature multiply review cycles.
> 3. **Tooling paralysis** — adopting complex monorepo tooling the team can't maintain.
>
> **Which option avoids more failure modes?**
>
> Monorepo avoids #1 and #2 decisively. Polyrepo only avoids #3.
>
> **But #3 is real.** Margin of safety says: start monorepo with *minimal* tooling. GitLab CI with path-based triggers (native). Add sophisticated tooling only when build times force it.
>
> **Opportunity cost:** polyrepo now means retrofitting a monorepo later when shared-model pain becomes unbearable — far more expensive than starting unified.
>
> **Confidence:** High
> **Where I May Be Wrong:** If the 4 services are truly independent (no shared models in practice), polyrepo's isolation benefits outweigh coordination costs.

#### user-advocate (Developer Experience)

> **Monorepo daily experience:** Engineer opens one repo, sees everything. Shared models change once, all consumers update atomically. IDE search works across boundaries. Friction: unfamiliar tooling for 2-3 weeks.
>
> **Polyrepo daily experience:** Familiar git workflows from day one. But shared data models mean publishing packages, managing versions, debugging "which version is deployed where?" weekly. Cross-service refactors span 4+ repos. Higher cognitive load disguised as simplicity.
>
> **My position:** Monorepo. The 2-week tooling learning curve is a one-time cost. The polyrepo coordination tax is permanent and scales with team size.
>
> **Confidence:** High
> **Where I May Be Wrong:** If services diverge significantly in lifecycle/ownership, monorepo coupling becomes a liability.

### Phase 2: Anonymized Peer Review (Summary)

Responses were shuffled to A-F. Key peer review findings:

- **Strongest response votes:** Deriver (3 votes) — praised for the O(1) vs O(N) cost function framing
- **Biggest blind spot flagged:** Systems-Mapper's coupling loop concern — strongest unique insight
- **What ALL missed:** CODEOWNERS enforcement, BFF-to-frontend type sharing, testing strategy

---

### Phase 3: Chairman Verdict (Final Output to User)

## Council Verdict: Monorepo vs Polyrepo

### Consensus
- All 6 advisors recommend monorepo — zero dissent on direction
- Shared data models + shared UI library create coupling edges that demand atomic changes
- 12 engineers is small enough for trunk-based development
- Tooling unfamiliarity is one-time (2-3 weeks); polyrepo coordination tax is permanent
- Start minimal (pnpm workspaces + path-based CI), grow into Nx/Turborepo when forced

### Clash Points
- **Tooling timing:** Shipper says "Turborepo takes an afternoon" vs Systems-Mapper/Inverter warning against complex tooling day one. Minor disagreement on *when*, not *whether*.
- **Coupling risk:** Systems-Mapper flags a delayed balancing loop (CI bottleneck at 6-9 months). Others treat coupling as net-positive given the topology.

### Blind Spots Caught
- No advisor addressed **CODEOWNERS enforcement** — critical at even 12 engineers
- No advisor modeled **BFF-to-frontend type sharing** — another coupling edge
- **Testing strategy** unaddressed: without affected-test detection, every PR runs all tests

### Strongest Dissent
Systems-Mapper's warning: the productivity loop masks a delayed CI bottleneck at 6-9 months. Not a reason to choose polyrepo — but a reason to invest in CI caching and affected-only test runs from week one.

### Confidence: 6/6 advisors align — HIGH
Unanimous. The shared dependency topology makes this structurally clear.

### The Recommendation
**Monorepo.** Start minimal: pnpm workspaces, GitLab CI with path-filtered pipelines, no Nx/Turborepo until build times exceed 5 minutes.

### First Move
Set up the monorepo scaffold with workspace packages for each service, shared UI library, shared data models, and BFF layer. Configure GitLab CI with `rules:changes` triggers. Prove the pipeline works empty before writing real code.

## Example 3: Follow-Up Protocol

After receiving a verdict:

**Expand:**
```
User: expand on the systems-mapper's point about the CI bottleneck loop
```
Re-presents the full analysis with detail: build-time growth -> developer frustration -> workarounds -> quality degradation -> retries. Recommends CI time budget from day one.

**Challenge:**
```
User: challenge the verdict with: our CTO wants each team to own their deployment independently
```
Re-runs synthesis with new constraint. Result: monorepo for code + independent per-service deployment pipelines. Code topology and deployment topology are separable concerns.

**Reweight:**
```
User: weight toward risk
```
Re-synthesizes emphasizing tail-watcher and inverter perspectives. May surface concerns about monorepo's single-point-of-failure CI.

**Re-run:**
```
User: re-run with innovation profile
```
Fresh council with questioner, subtractor, reframer, taxonomist, deriver, inverter. Likely challenges the premise: "Do you actually need 4 microservices, or is that premature decomposition?"
